
## 프로그래밍 요구 사항 (Kotlin 스타일)

* **Google Kotlin Style Guide**를 준수하여 코드를 작성한다.
* **indent(인덴트, 들여쓰기) depth는 2까지만 허용한다.**
* **삼항 연산자(조건 표현식)** 대신 `when`, `if` 표현식으로 명확하게 처리한다.
* **함수는 한 가지 일만 수행**하도록 작게 분리한다.
* **함수(또는 메서드)의 길이**는 **15라인을 초과하지 않는다.**
* **else 예약어 대신 early return**을 사용한다.
* **클래스와 함수는 단일 책임 원칙(SRP)** 을 따른다.
* **sealed class / enum class** 를 활용하여 상태나 이벤트를 표현한다.
* **data class** 를 사용해 불변(immutable) 데이터 구조를 유지한다.
* **코틀린 컬렉션 함수(map, filter, forEach 등)** 를 적극 활용한다.
* **코루틴(Coroutine)** 을 사용할 경우, 명확한 Job 관리와 예외 처리를 구현한다.
* **의존성 주입(DI)** 은 생성자 주입을 기본으로 한다.
* **테스트 코드(JUnit5, AssertJ)** 로 모든 핵심 비즈니스 로직을 검증한다.
* **함수형 프로그래밍 스타일**을 지향하되, 가독성을 우선시한다.
* **불필요한 클래스 상속, 전역 변수, static 접근은 금지한다.**
* **nullable 타입은 최소화하고**, `?.`, `?:`, `!!` 사용 시 명확한 의도를 주석으로 명시한다.

## 커밋 메시지 규칙

* **AngularJS Git Commit Message Conventions**를 따른다.
* 커밋 메시지는 다음 구조를 따른다.

```text
type: subject

body (optional)

footer (optional)
```

### Type 종류

| Type       | 설명                             |
| ---------- | ------------------------------ |
| `feat`     | 새로운 기능 추가                      |
| `fix`      | 버그 수정                          |
| `docs`     | 문서 수정                          |
| `style`    | 코드 포맷팅, 네이밍 변경 등 (코드 로직 변경 없음) |
| `refactor` | 코드 리팩토링                        |
| `test`     | 테스트 코드 추가/수정                   |
| `chore`    | 빌드, 설정, 의존성 수정 등               |

### 작성 규칙

| 영역         | 핵심 포인트                        |
| ---------- | ----------------------------- |
| **커밋 단위**  | 한 가지 의도만 담는다                  |
| **메시지 규칙** | `type: subject` 구조 유지         |
| **브랜치 전략** | feature/fix/release 기준 명확히 구분 |
| **협업 전략**  | rebase로 커밋 정리 후 PR            |
| **리뷰 효율성** | 작은 커밋이 좋은 커밋이다                |


### 예시

```text
feat: 사건 발생 시 주가 변동 로직 추가

refactor: MarketEventBus의 이벤트 중복 방지 로직 개선

test: StockService의 가격 변동 테스트 추가
```
